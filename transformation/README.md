# Data Transformation

This folder contains the transformation layer of the **CarePlus Support Analytics Pipeline**.

The goal of this stage is to convert raw ingested data into a clean, structured, and analytics-ready format before loading it into the data warehouse.

Two different AWS services were used during this phase:

* **AWS Lambda** for support log transformation
* **AWS Glue** for support ticket transformation

This approach was intentionally chosen to gain hands-on experience with multiple AWS ETL services and understand their use cases.

# Transformation Overview

The raw datasets ingested into Amazon S3 undergo a series of cleaning and standardization steps before being stored in the Processed Zone.

Common transformation activities include:

* Data parsing
* Data cleansing
* Data validation
* Data type standardization
* Duplicate removal
* Error handling
* Conversion to Parquet format

The final transformed datasets are stored in Amazon S3 and later loaded into Amazon Redshift for analytics and reporting.

---

# Support Logs Transformation (AWS Lambda)

The support logs dataset consists of unstructured log files generated by backend systems.

To process these logs:

1. Raw log files are uploaded to Amazon S3.
2. An S3 event notification is triggered.
3. AWS Lambda executes the transformation logic.
4. Log entries are parsed and converted into structured records.
5. Data is cleaned and standardized.
6. Output is stored as Parquet files in the S3 Processed Zone.

### Why AWS Lambda?

AWS Lambda was selected because:

* The files are relatively small.
* Processing is event-driven.
* No infrastructure management is required.
* It provides a lightweight serverless solution for log processing.

# Support Tickets Transformation (AWS Glue)

The support tickets dataset originates from a structured MySQL database and is ingested into Amazon S3 as CSV files.

To process these files:

1. Ticket CSV files are uploaded to Amazon S3.
2. An S3 event notification triggers an AWS Lambda function.
3. The Lambda function starts an AWS Glue ETL job.
4. AWS Glue performs the required transformations.
5. Data is converted into Parquet format.
6. Output is stored in the S3 Processed Zone.

### Why AWS Glue?

AWS Glue was selected because:

* It is designed specifically for ETL workloads.
* It can efficiently process structured datasets.
* It provides both visual and script-based ETL development.
* It integrates seamlessly with the AWS analytics ecosystem.

# Event-Driven Architecture

The transformation layer follows an event-driven design.

Whenever a new file arrives in the S3 Raw Zone, an event notification automatically initiates the appropriate transformation workflow.

```text
New File Uploaded
        ↓
Amazon S3 Raw Zone
        ↓
S3 Event Notification
        ↓
Transformation Service
(Lambda or Glue)
        ↓
Amazon S3 Processed Zone
```

This eliminates the need for manual execution and enables automated processing of newly ingested data.

---

# Why Parquet?

Both transformation pipelines generate output in **Parquet format**.

Parquet was chosen because it:

* Provides efficient columnar storage.
* Reduces storage requirements.
* Improves query performance.
* Integrates well with Athena and Redshift.
* Is widely used in modern data engineering workflows.
